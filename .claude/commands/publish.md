# /publish - drafts を記事に昇格する

`tech-library-drafter` が `drafts/<YYYY-MM>/` に貯めた下書きを、**人がレビューして選び**、
このリポジトリの正式な記事（`languages/` `concepts/` `tools/` `error/` `daily/`）へ昇格させるコマンド。

- **`/log`** … 貼り付けた Markdown を直接記事化する（従来どおり、このコマンドとは別物）
- **`/publish`** … 既に drafts に貯まっている下書きから昇格する（このコマンド）
- **README は触りません**。README の更新と push は `/push` で行います。

**重要: 既存記事の本文を Read で読まないこと。** 類似チェックは bash + frontmatter 抽出方式で行う（トークン節約）。
昇格対象の draft 本文だけは Read してよい（変換に必要なため）。

## ユーザー入力

$ARGUMENTS

引数がある場合の解釈:
- 月（`2026-07` 等）→ その月の drafts だけを対象にする
- draft ファイルパス → その1件だけを対象にする
- 空 → すべての月の未昇格 draft を対象にする

---

## あなた（Claude）が行う処理

### 1. 未昇格 draft の一覧を bash で抽出

```bash
for f in $(find drafts -name "*.md" -type f 2>/dev/null | sort); do
  echo "===FILE==="
  echo "$f"
  # frontmatter（status/source/date/tags）
  awk '/^---$/{c++; next} c==1{print} c==2{exit}' "$f"
  # frontmatter 後の最初の H1（タイトル）と一行サマリ
  awk 'f&&/^# /{print "H1:" substr($0,3); getline t; while(t==""){getline t}; print "SUMMARY:" t; exit} /^---$/{c++} c>=2{f=1}' "$f"
done
```

出力から各 draft の `path` / `status` / `date` / `source` / `tags` / `H1`(タイトル) / `SUMMARY`(一行サマリ) を得る。

**フィルタ**: `status: published` の draft は**除外**（昇格済み）。`status: draft` のものだけを候補にする。
引数で月やパスが指定されていれば、その範囲にさらに絞る。

候補が0件なら「昇格できる下書きはありません（すべて published 済み、または drafts が空です）」と伝えて終了。

### 2. 一覧を提示してユーザーに選ばせる（人のレビュー点）

候補を **番号付きリスト** で提示する。開かずに中身が判断できるように、1件ずつ次を1行で見せる:

```
昇格できる下書き（未 published）:

 1. [2026-07-03] Apollo は id と __typename が揃えば正規化キャッシュを自動更新する
    source: 実務(Apollo) / tags: apollo, graphql, cache
    drafts/2026-07/2026-07-03-apollo-cache-normalization.md
 2. [2026-07-05] ...
```

そのうえで、**どれを昇格するか** を聞く（複数選択可）。候補が2〜4件なら `AskUserQuestion` の
multiSelect を使ってよいが、件数が多い場合は上のリストを見せて「昇格する番号を教えてください
（例: `1,3,4` / `all` / `none`）」と自由入力で受ける。

- `none` または選択なし → 何も昇格せず終了
- 選ばれた draft だけを、以降の手順3を **1件ずつ順番に** 処理する

**勝手に全部昇格しない。** 必ずユーザーの選択を待つ。

### 3. 選ばれた draft を1件ずつ昇格する

各 draft について、以下を順に行う。

#### 3-0. draft 本文を Read する

昇格対象の draft ファイルのみ Read する（変換に本文が要るため）。

#### 3-1. 機密情報チェック（最優先・公開リポジトリ前提）

drafter 側でサニタイズ済みのはずだが、**公開前の最後の関門**として必ず再確認する。
本文・`source`・タイトル・タグ・（後で決める）スラッグに以下が1つでも残っていたら、
その draft の昇格を**中止**し、ユーザーに「サニタイズが必要」と伝えて次の draft へ進む。

- 実プロジェクト名・製品名・社内コードネーム（例: 秘匿対象の内部プロジェクト名）
- 社名・顧客名・取引先名・個人名・メールアドレス・アカウント名
- 社内 URL・内部ホスト名・エンドポイントのパス・IP・ポート・issue 参照（`PROJECT issue #NN` 等）
- API キー・トークン・パスワード・シークレット・`.env` の値
- 実在の DB 名・テーブル名・カラム名・実コードの固有識別子（実関数名・実モデル名など）
- 未公開の設計・ビジネスロジック・非公開仕様

少しでも「公開して困るかも」と感じたら中止して報告する。判断に迷ったら昇格しない。

#### 3-2. カテゴリ分類

draft の内容から、昇格先カテゴリを1つ決める（`/log` と同じ基準）。

| カテゴリ | 保存先 | 該当例 |
| --- | --- | --- |
| 特定言語の知見 | `languages/<lang>/` | TypeScript の型推論、Ruby のキーワード引数 など |
| 言語横断の概念 | `concepts/<topic>/` | DDD、認証フロー、設計パターン、アルゴリズム など |
| ツール / インフラ | `tools/<tool>/` | React、Rails、Apollo、GraphQL、Docker、Git など |
| エラー・障害解析 | `error/<topic>/` | エラー原因レポート、障害ポストモーテム |
| 雑メモ・未整理 | `daily/` | 上記に明確に分類できないもの |

`<lang>` `<topic>` `<tool>` は kebab-case 英小文字。ディレクトリが無ければ作る。
**予定タグ集合**（2〜5個・先頭はカテゴリ名）も決めておく（次の類似チェックで使う）。

#### 3-3. 類似記事チェック（重複乱立を防ぐ）

**デフォルトは「新規作成」。** 明確な重複・類似が見つかったときだけ、まとめる選択肢を提案する。

`/log` と同じ方式で、既存記事の frontmatter だけを抽出して比較する（本文は読まない）:

```bash
for f in $(find languages concepts tools daily error -name "*.md" -type f 2>/dev/null | sort); do
  echo "===FILE==="
  echo "$f"
  awk '/^---$/{c++; next} c==1{print} c==2{exit}' "$f"
done
```

予定タグ集合・予定保存先サブディレクトリと各既存記事を比較し、類似候補を集める:

| 判定 | 条件 |
| --- | --- |
| 強い類似 | 予定保存先サブディレクトリ（例: `tools/apollo-client/`）に既存記事がある |
| 中程度の類似 | タグの共通数が2個以上 |
| テーマ類似 | タイトルの主要キーワード（カタカナ語・英単語）が2つ以上一致 |

- **候補が0件 → 何も聞かず、そのまま新規作成（3-4）へ進む。** これが基本の流れ。
- **候補が1件以上あるときだけ** `AskUserQuestion` で「まとめるか、新規で作るか」を提案する。
  類似度の高い順に最大3件を候補として見せ、選択肢は次の順で提示する（**新規が既定・先頭**）:
  - **C: このまま新規作成（推奨）** ← 先頭・デフォルト。迷ったらこれ。
  - A: `<最有力候補のpath>` にまとめて追記
  - B: 別候補にまとめて追記（他に候補があれば）

  提案時は「◯◯と内容が重複していそうです。まとめますか？ それとも新規で作りますか？」のように、
  **どの既存記事とどう重複していそうか**を一言添える。勝手にマージも新規もしない（選択を待つ）。

#### 3-4. draft スキーマ → 記事スキーマへ変換

draft（`status/source/date/tags` + 背景/現象/原因/解決/学び/参考）を、記事の形へ集約する。

**frontmatter**（記事側の形式）:

```yaml
---
title: <draft の H1 見出しテキストをそのまま日本語タイトルに>
date: <draft の date をそのまま使う（学びが起きた日。昇格日ではない）>
tags: [<カテゴリ名を先頭に。draft の tags を流用・調整>]
source: <draft の source をそのまま（実務(Rails) / 個人開発 等の一般カテゴリ）>
---
```

- `title` は draft の `# 見出し` から取る（draft は frontmatter に title を持たないため）。
- `date` は **draft の date を維持**する（README の並びが「学んだ順」になる）。
- 先頭タグはカテゴリ名（react / rails / typescript など）に揃える。

**本文**（`概要` → `サンプルコード` → `ポイント` に集約する）:

```markdown
## 概要

<draft の 一行サマリ + 背景 を 1〜3 行に要約。何についての記事かを述べる>

## サンプルコード

<draft の「解決」にある最小の汎用サンプルを載せる。コードが無ければこのセクションごと省略>
<コードブロックには必ず言語タグ（```ts, ```rb, ```bash 等）>

## ポイント

- <draft の 原因 / 学び・応用 から要点を箇条書きに>
- <2〜5 項目>
```

任意で、draft の `## 参考` に**確認できた一次情報 URL** があれば、本文末尾に
`## 参考` として残してよい（URL が確かなものだけ。無ければ付けない）。

**変換の哲学（厳守）**:
- draft に**書かれている事実だけ**を要約・再構成する。**新しい知識・解説・コードを足さない**。
- 言い換えで意味を変えない。不確かな点は書かない。
- draft の 6 セクションを 3 セクションに畳むのは「要約と統合」であって「創作」ではない。
- コードは draft の汎用サンプルをそのまま使う（実コードの再現・拡張はしない）。

#### 3-5. 保存先ファイル名の決定（新規の場合）

形式: `<カテゴリ保存先>/YYYY-MM-DD-<kebab-case-slug>.md`

- 日付は draft の `date`（3-4 と揃える）。
- スラッグは draft ファイル名の日付以降の部分を流用してよい（既に具体的な英語スラッグのはず）。
  カテゴリに合わなければ内容が分かる具体語で付け直す。固有名・社名は入れない。
- 例: `tools/apollo-client/2026-07-03-apollo-cache-normalization.md`
- 同じディレクトリに同名ファイルがあれば、上書きせず末尾に `-2`, `-3` を付ける
  （`Test-Path <path>` / `[ -e <path> ]` で存在確認し、空くまで番号を上げる）。

**追記（3-3 で「追記」を選んだ）の場合**: ファイル名は新規に作らず、既存記事を Read → 末尾に
`## 追記：<小タイトル>` セクションとして 3-4 の本文を足す。frontmatter の `date` を
**draft の date に更新**（最終更新日）、`title`/`source` は既存のまま、新キーワードが無ければ `tags` に追加。

#### 3-6. 記事ファイルの保存

決めたパスに記事を書き込む（追記モードなら既存記事を Edit）。ディレクトリが無ければ作る。
`drafts/` 以外の既存記事フォルダには、昇格先以外を触らない。

#### 3-7. 元の draft を published にマークして残す

昇格した draft は**削除せず**、frontmatter を更新して痕跡と昇格先を残す（二重昇格の防止・追跡用）:

- `status: draft` → `status: published` に変更
- `promoted_to: <保存した記事の相対パス>` を追記（既存キーの後ろに1行）

draft の本文はそのまま残す。

#### 3-8. コミット

記事の追加/更新と draft の status 変更を **1つのコミット** にまとめる:

```bash
git add <保存した記事パス> <昇格した draft パス>
git commit -m "<prefix>: <記事タイトル>"
```

プレフィックス:
- `add:` … 新規記事として昇格（3-3 で新規 / 候補なし）
- `update:` … 既存記事へ追記して昇格（3-3 で追記）

（1回の `/publish` で複数 draft を昇格する場合は、draft ごとに 3-0〜3-8 を繰り返し、
**draft 1件 = 1コミット** にする。どれがどの記事になったか履歴で追えるようにする。）

### 4. 完了報告

昇格した全 draft をまとめて報告する:

```
✅ N 件の下書きを記事に昇格しました

 • add:    <記事パス1>   ← <draft パス1>
 • update: <記事パス2>   ← <draft パス2>

⏭  スキップ: <サニタイズ中止・未選択などがあれば理由付きで>
ℹ  README とリモートへの反映は /push で行ってください
```

## まとめ：このコマンドの役割

- **人が選んだ** draft だけを、既存記事の作法（分類・類似チェック・記事テンプレ）に沿って昇格する
- **昇格は新規作成がデフォルト**。内容が重複していそうなときだけ「まとめる／新規」を提案する（新規が既定）
- draft の**事実だけ**を `概要 / サンプルコード / ポイント` に集約する（知識は足さない）
- 公開リポジトリ前提で、昇格の**直前に必ずサニタイズを再確認**する（最後の関門）
- 昇格した draft は `status: published` + `promoted_to:` で残し、追跡可能にする
- README・push はこのコマンドの責務外（`/push` に委ねる）
