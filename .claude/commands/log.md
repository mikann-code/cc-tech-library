# /log - 技術メモを記録する

ユーザーが直前に貼り付けた、または引数として渡した内容（Markdown / コードスニペット / 技術メモ）を、このリポジトリのナレッジ記事として保存してください。

**README は触りません**。README の更新と push は `/push` コマンドで行います。

## ユーザー入力

$ARGUMENTS

---

## あなた（Claude）が行う処理

以下の手順を **必ず順番に** 実行してください。

### 1. 機密情報チェック（最優先）

入力内容に以下が含まれていたら、**保存を中止して警告**してください。
- 会社名・サービス名・プロダクト名（パブリックな OSS 名は除く）
- 顧客名、個人名（一般的な著者名は除く）
- API キー、トークン、パスワード、内部 URL、IP アドレス
- 社内独自の設計、内部仕様

該当がある場合: 「機密情報が含まれている可能性があります: <該当箇所> 抽象化してから再度貼り付けてください」と返して終了。

### 2. 内容の分類

入力内容を読んで、以下のどのカテゴリに最も当てはまるかを判断してください。

| カテゴリ | 保存先 | 該当例 |
| --- | --- | --- |
| 特定言語の知見 | `languages/<lang>/` | TypeScript の型推論、Python のデコレータ、Go の goroutine など |
| 言語横断の概念 | `concepts/<topic>/` | DDD、認証フロー、設計パターン、データ構造、アルゴリズムなど |
| ツール / インフラ | `tools/<tool>/` | Docker、Git、CI/CD、AWS、Kubernetes など |
| 雑メモ・未整理 | `daily/` | 上記に明確に分類できない、後で整理するもの |

`<lang>` `<topic>` `<tool>` は kebab-case の英小文字（例: `typescript`, `domain-driven-design`, `docker`）。該当ディレクトリがなければ作成してください。

### 3. ファイル名の決定

形式: `YYYY-MM-DD-<kebab-case-title>.md`

- 日付は今日の日付（`date +%F` などで取得）
- タイトルは内容を 3〜6 単語で要約した kebab-case
- 例: `2026-05-06-typescript-conditional-types.md`

同じ日に同名のファイルがあれば末尾に `-2`, `-3` を付ける。

### 4. Frontmatter の付与

ファイルの先頭に以下の YAML frontmatter を付ける。

\`\`\`yaml
---
title: <人間が読める日本語タイトル>
date: <YYYY-MM-DD>
tags: [<lang/topic/tool>, <関連キーワード1>, <関連キーワード2>, ...]
source: <出典が分かれば。インターン x 日目、書籍名、など。不明なら省略可>
---
\`\`\`

タグは 2〜5 個。最初のタグはカテゴリ名（typescript / docker など）に揃える。

### 5. 本文の整形

- 入力がそのまま記事になるなら、frontmatter を付けるだけでよい
- 箇条書きの羅列なら、適度に見出し（`##`）を補って読みやすくする
- コードブロックには言語タグを付ける
- 内容を勝手に**水増ししない**。ユーザーが書いていない情報を補足追記しない（誤った知識を残さないため）

### 6. ファイル保存

決定したパスにファイルを書き込む。書き込み先のディレクトリが存在しなければ作成する。

### 7. Git コミット

以下を順に実行する。

\`\`\`bash
git add <保存したファイル>
git commit -m "add: <タイトル>"
\`\`\`

コミットメッセージは以下のプレフィックスを使い分ける。
- \`add:\` 新規記事
- \`update:\` 既存記事の追記・修正
- \`refactor:\` 整理・リネーム

### 8. 完了報告

ユーザーに以下を簡潔に報告する。

\`\`\`
✅ 保存しました
📁 <保存先パス>
🏷  <付けたタグ>
💬 <コミットメッセージ>
ℹ  README とリモートへの反映は /push で行ってください
\`\`\`

---

## 注意

- 保存前に必ず機密情報チェックを行うこと
- 既存ファイルと内容が酷似する場合は、新規作成ではなく既存ファイルへの追記を提案する
- 分類に迷う場合はユーザーに確認する
- ユーザーが書いていない情報を勝手に補わない（**事実の改変はしない**）
- **README は絶対に変更しないこと（/push の役割）**
