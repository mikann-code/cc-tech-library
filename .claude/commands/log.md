# /log - 技術メモを記録する

ユーザーが直前に貼り付けた、または引数として渡した内容（Markdown / コードスニペット / 技術メモ）を、このリポジトリのナレッジ記事として保存してください。

**README は触りません**。README の更新と push は `/push` コマンドで行います。

## ユーザー入力

$ARGUMENTS

---

## あなた（Claude）が行う処理

以下の手順を **必ず順番に** 実行してください。

### 1. 機密情報チェック（最優先）

入力内容に以下が含まれていたら、**保存を中止して警告**してください。
- 会社名・サービス名・プロダクト名（パブリックな OSS 名は除く）
- 顧客名、個人名（一般的な著者名は除く）
- API キー、トークン、パスワード、内部 URL、IP アドレス
- 社内独自の設計、内部仕様

該当がある場合: 「機密情報が含まれている可能性があります: <該当箇所> 抽象化してから再度貼り付けてください」と返して終了。

### 2. 内容の分類

入力内容を読んで、以下のどのカテゴリに最も当てはまるかを判断してください。

| カテゴリ | 保存先 | 該当例 |
| --- | --- | --- |
| 特定言語の知見 | `languages/<lang>/` | TypeScript の型推論、Python のデコレータ、Go の goroutine など |
| 言語横断の概念 | `concepts/<topic>/` | DDD、認証フロー、設計パターン、データ構造、アルゴリズムなど |
| ツール / インフラ | `tools/<tool>/` | Docker、Git、CI/CD、AWS、Kubernetes など |
| エラー・障害解析 | `error/<topic>/` | エラー原因レポート、障害ポストモーテム |
| 雑メモ・未整理 | `daily/` | 上記に明確に分類できない、後で整理するもの |

`<lang>` `<topic>` `<tool>` は kebab-case の英小文字（例: `typescript`, `domain-driven-design`, `docker`）。該当ディレクトリがなければ作成してください。

合わせて **予定タグ集合**（2〜5 個、最初のタグはカテゴリ名）も決めておく。手順 3 で使う。

### 3. 類似記事チェック（重要 — 重複乱立を防ぐ）

似た記事が量産されないように、**保存前に必ず** 既存記事との類似度をチェックする。

#### 3-1. 全記事の frontmatter を抽出（本文は読まない）

`/push` と同じ方式で title + tags + path を一括取得する:

```bash
for f in $(find languages concepts tools daily error -name "*.md" -type f 2>/dev/null | sort); do
  echo "===FILE==="
  echo "$f"
  awk '/^---$/{c++; next} c==1{print} c==2{exit}' "$f"
done
```

#### 3-2. 類似度の判定基準

これから書こうとしている記事の **予定タグ集合**（手順 2 で決めたもの）と **予定保存先サブディレクトリ** を、各既存記事と比較し、以下のいずれかに該当する記事を「類似候補」として集める:

| 判定 | 条件 |
| --- | --- |
| 強い類似 | 予定保存先サブディレクトリ（例: `tools/graphql/`）に既存記事がある |
| 中程度の類似 | タグの共通数が 2 個以上 |
| テーマ類似 | タイトルの主要キーワード（カタカナ語・英単語）が 2 つ以上一致 |

該当記事を **類似度の高い順に最大 3 件** まで集める。

#### 3-3. 類似候補があれば必ずユーザーに確認

候補が 1 件以上あれば、`AskUserQuestion` ツール（または同等の確認手段）で必ず聞く。**勝手にマージしたり、勝手に新規作成したりしない**。

提示する選択肢の例:
- A) `<path1>` に追記する（共通タグ: ...）
- B) `<path2>` に追記する（共通タグ: ...）
- C) このまま新規ファイルとして作る

候補が 0 件なら、何も聞かずに手順 4 へ進む（無音）。

#### 3-4. 分岐

**「追記」が選ばれた場合**:
1. 該当ファイルを Read で読み込む
2. 既存記事の末尾に、`##` レベルの新セクション（例: `## 追記：<小タイトル>`）として今回の内容を追加する
3. frontmatter の `date` を **今日の日付に更新する**（最終更新日を表す）。ファイル名・`title`・`source` は元のまま
4. 新内容で扱う重要キーワードが既存 `tags` に無ければ、frontmatter の `tags` に **追加** する（既存タグは消さない）
5. 本文整形ルール（手順 6 と同等）は適用する — コードブロックには言語タグ、捏造禁止など
6. コミットメッセージは `update: <既存記事タイトルの要約>` プレフィックス
7. **手順 4（ファイル名決定）と手順 5（frontmatter 新規作成）はスキップ** し、手順 7（ファイル保存）と手順 8（commit）へ進む

**「新規」が選ばれた場合**: そのまま手順 4 以降に進む。

### 4. ファイル名の決定（新規作成時のみ）

形式: `YYYY-MM-DD-<kebab-case-title>.md`

- 日付は今日の日付（`date +%F` などで取得）
- タイトルは内容を 3〜6 単語で要約した kebab-case
- 例: `2026-05-06-typescript-conditional-types.md`

同じ日に同名のファイルがあれば末尾に `-2`, `-3` を付ける。

### 5. Frontmatter の付与（新規作成時のみ）

ファイルの先頭に以下の YAML frontmatter を付ける。

```yaml
---
title: <人間が読める日本語タイトル>
date: <YYYY-MM-DD>
tags: [<lang/topic/tool>, <関連キーワード1>, <関連キーワード2>, ...]
source: <出典が分かれば。インターン x 日目、書籍名、など。不明なら省略可>
---
```

タグは 2〜5 個。最初のタグはカテゴリ名（typescript / docker など）に揃える。

### 6. 本文を「概要 → サンプルコード → ポイント」のテンプレートに整形

frontmatter の後（または追記モードでは末尾に追加するセクション内で）、以下の構成で本文を組み立てる。

```markdown
## 概要

<入力内容から抜き出した、何についての記事かの 1〜3 行の説明>

## サンプルコード

<入力にコードが含まれていた場合のみこのセクションを作る>
<コードブロックには必ず言語タグを付ける（例: ```ts, ```py, ```bash）>
<複数のコードがあれば全部残す。コードの前後にユーザーが書いていた説明文があればそれも一緒に残す>

## ポイント

- <入力内容から抜き出した重要な点を箇条書きで>
- <ユーザーが書いた内容のみを根拠にする>
- <2〜5 項目程度>
```

#### 重要な整形ルール

**コードがない場合**:
「サンプルコード」セクション**ごと省略**する。空のセクションを作らないこと。コードを **絶対に捏造しない**。

**コードがある場合**:
- コードブロックには必ず言語タグを付ける（例: \`\`\`ts, \`\`\`py, \`\`\`bash, \`\`\`yaml, \`\`\`sh）
- 入力に複数のコードブロックがあれば、それぞれ独立した小見出し（例: `### 基本形`, `### 応用例`）で並べてもよい
- コードの直前にユーザーが書いていた1〜2行の文脈説明があれば、それも一緒に残す

**ポイントの抜き出し方**:
- ユーザーの入力に書かれている内容のみを根拠にする
- 「〜という意味」「〜と同じ」のような **言い換えはしない**（誤った知識を残すリスク）
- 入力に箇条書きがあればそれを優先採用
- 散文の場合は、要点を抽出して箇条書き化する

**「概要」の書き方**:
- ユーザーが冒頭に書いた1〜3行があればそれをそのまま使う
- なければ、入力全体を見て **何の話題なのかだけ** を1行で書く（深掘り解説はしない）

#### 絶対にやってはいけないこと

- ❌ ユーザーが書いていない知識を **補足追記** する（誤情報リスク）
- ❌ ユーザーが書いていないコード例を **追加生成** する
- ❌ ユーザーが書いた内容を **言い換えて意味を変える**
- ❌ 「ちなみに〜」「補足: 〜」のような Claude の知識を混ぜる

迷ったら **「ユーザーが書いた内容だけを並び替えて読みやすくする」** に徹すること。

### 7. ファイル保存

決定したパスにファイルを書き込む（追記モードなら既存ファイルを Edit）。書き込み先のディレクトリが存在しなければ作成する。

### 8. Git コミット

以下を順に実行する。

```bash
git add <保存したファイル>
git commit -m "<prefix>: <タイトル>"
```

コミットメッセージのプレフィックスを使い分ける:
- `add:` 新規記事（手順 3 で「新規」を選択 / 候補なし）
- `update:` 既存記事への追記（手順 3 で「追記」を選択）
- `refactor:` 整理・リネーム

### 9. 完了報告

ユーザーに以下を簡潔に報告する。

**新規作成時**:
```
✅ 保存しました
📁 <保存先パス>
🏷  <付けたタグ>
💬 <コミットメッセージ>
ℹ  README とリモートへの反映は /push で行ってください
```

**追記時**:
```
✅ 既存記事に追記しました
📁 <追記先パス>
➕ 追加したセクション: <小タイトル>
💬 <コミットメッセージ>
ℹ  README とリモートへの反映は /push で行ってください
```

---

## まとめ：整形の哲学

このコマンドの役割は **「ユーザーが書いた知識を構造化して保存する」** だけです。
**新しい知識を生成することではありません。**

ユーザーの入力に対して
- ✅ セクション分け / 言語タグ付け / kebab-case化 などの **形式整形** はする
- ✅ 似た既存記事を見つけたら **追記提案** をする（重複乱立の防止）
- ❌ 知識の **追加・解説・補完** はしない

これにより、後から読み返した時に「あの時自分が学んだことそのもの」が残る、信頼できる記録になります。
