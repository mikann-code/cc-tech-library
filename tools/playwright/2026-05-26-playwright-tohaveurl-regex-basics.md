## 概要

Playwright の `expect(page).toHaveURL()` で使う正規表現の基本まとめ。文字列マッチと正規表現マッチの使い分け、`\/` のエスケープ、`\d+` の意味、デフォルトの「含む」マッチの挙動などを整理する。

## サンプルコード

### 1. 2つの書き方

`toHaveURL` は **文字列** と **正規表現** の両方を受け取れる。

```ts
// 文字列: 完全一致
await expect(page).toHaveURL("/dashboard");

// 正規表現: パターンマッチ
await expect(page).toHaveURL(/\/public-events\/\d+/);
```

| 書き方 | 使うとき | 例 |
| --- | --- | --- |
| 文字列 `"..."` | URL が **完全に固定** | `/dashboard`、`/login` |
| 正規表現 `/.../` | URL の **一部が可変** | `/public-events/{id}` |

**なぜ使い分けるか**

- ログイン後の `/dashboard` は **常に同じ URL** → 文字列で OK
- イベント詳細の `/public-events/{id}` は ID が seed のたびに変わる → 正規表現が必要
  - 1回目の seed: `/public-events/1`
  - 2回目の seed: `/public-events/2`
  - ID は AUTO INCREMENT で毎回変動

### 2. 正規表現の基本構造

```
/\/public-events\/\d+/
```

ブロックに分けると:

```
[/]  [\/]  [public-events]  [\/]  [\d+]  [/]
```

| パーツ | 役割 | URL 側の対応 |
| --- | --- | --- |
| `[/]` 先頭 | 「これ正規表現だよ」の囲い | （JS の文法上の記号） |
| `[\/]` | 本物のスラッシュ | URL の 先頭の `/` |
| `[public-events]` | リテラル文字 | URL の `public-events` |
| `[\/]` | 本物のスラッシュ | URL の 真ん中の `/` |
| `[\d+]` | 数字1個以上のパターン | URL の ID 部分 |
| `[/]` 末尾 | 「これ正規表現だよ」の囲い | （JS の文法上の記号） |

**文字列との対比**

```ts
"hello"   // 文字列: " で囲む
/hello/   // 正規表現: / で囲む
```

外側の囲いは「これ何のデータ?」を JS に伝えるしるし。文字列の `"..."` と同じ役割。

### 3. エスケープ（`\/`）の話

正規表現リテラルは `/` で始まり `/` で終わる。中で `/` を書きたいときは `\/` とエスケープが必要。

```
//public-events/      ❌ 1個目の // が空っぽの正規表現と勘違いされる
/\/public-events\//   ✅ 本物の / だよと教える
```

**同じ仕組みは文字列にもある**

```
"彼は\"おはよう\"と言った"
//   ↑          ↑
//   └ \" で本物の " を表す
```

外と中で同じ記号を使うときはエスケープが必要、という共通ルール。

### 4. `\d+` の意味

| 記号 | 意味 |
| --- | --- |
| `\d` | 数字1文字（0〜9のどれか1個） |
| `+` | 直前のものが 1個以上 |
| `\d+` | 数字が1個以上連続している部分 |

**マッチ例**

| 入力 | `\d+` でマッチ? |
| --- | --- |
| `1` | ✓ |
| `123` | ✓ |
| `9999` | ✓ |
| `abc` | ✗ |
| 空文字 `""` | ✗ |

**他の量指定子**

| 書き方 | 意味 |
| --- | --- |
| `\d` | 数字ちょうど1個 |
| `\d+` | 数字 1個以上 |
| `\d*` | 数字 0個以上 |
| `\d{3}` | 数字 ちょうど3個 |
| `\d{2,4}` | 数字 2〜4個 |

### 5. デフォルトは「含む」マッチ

正規表現は ブラウザの Ctrl+F と同じで、デフォルトは「含むかどうか」のチェック。

```
/\/public-events\/\d+/
```

| URL | マッチ? | 理由 |
| --- | --- | --- |
| `/public-events/123` | ✓ | パターンが含まれる |
| `http://localhost:3000/public-events/123?ref=top` | ✓ | パターンがどこかに含まれる |
| `/events/123` | ✗ | `public-events` が含まれない |

**完全一致したいときは `^` と `$`**

```
/^\/public-events\/\d+$/
 ↑                    ↑
 └ 文字列の始まり      └ 文字列の終わり
```

| 記号 | 意味 |
| --- | --- |
| `^` | 文字列の始まり |
| `$` | 文字列の終わり |

### 6. `\` の二面性

`\` は **次の文字とセットで意味が決まる**。

| 組み合わせ | 種類 | 意味 |
| --- | --- | --- |
| `\/` | エスケープ | 特殊記号 `/` を「ただの文字」に格下げ |
| `\d` | メタ文字 | 普通の文字 `d` を「数字」という特殊な意味に格上げ |
| `\w` | メタ文字 | 英数字1文字 |
| `\s` | メタ文字 | 空白文字1文字 |
| `\\` | エスケープ | バックスラッシュそのもの |

ポイント: `\` 単体では意味を持たない。必ず次の1文字とセットで読む。

### 7. 実用上のチートシート

```ts
// よく使うパターン
await expect(page).toHaveURL("/login");                 // 完全一致
await expect(page).toHaveURL("/dashboard");             // 完全一致
await expect(page).toHaveURL(/\/public-events\/\d+/);   // 可変 ID
await expect(page).toHaveURL(/\/users\/\d+/);           // 可変 ID
await expect(page).toHaveURL(/login/i);                 // 大文字小文字無視
```

困ったら ChatGPT に「URL が `/xxx/数字` かチェックする正規表現を書いて」と聞けば出してくれるので、丸暗記は不要。

## ポイント

- `/.../` は正規表現の囲い。文字列の `"..."` と同じ役割
- 中で `/` を使うときは `\/` とエスケープ
- `\d+` = 「数字1個以上」、URL の可変 ID 部分のマッチに使う
- デフォルトは「含む」マッチ、完全一致したいなら `^` と `$`
- 固定 URL は文字列、可変 URL は正規表現 で使い分ける
