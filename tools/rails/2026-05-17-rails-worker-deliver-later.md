## 概要

Rails における Worker の役割と、メール送信を `deliver_now`（同期）/ `deliver_later`（非同期）で行うときの違い、Solid Queue（MySQL ベースのジョブキュー）を使った構成、なぜ `deliver_later` を選ぶのかの理由まで。

## サンプルコード

### deliver_now と deliver_later のコード上の見た目

```ruby
# 同期送信（その場で送る）
EmailChangeMailer.email_change_email(user).deliver_now

# 非同期送信（worker に任せる）
EmailChangeMailer.email_change_email(user).deliver_later
```

## 1. そもそも Worker とは

ざっくり言うと「裏でこっそり仕事をしてくれる別のプロセス」のこと。

普段の Rails アプリは「リクエストが来たら処理して返す」という流れですが、これだと 時間のかかる処理 が問題になります。そこで「時間がかかる仕事は別の人（worker）に任せて、本体はすぐ次のリクエストに対応する」という分業制にします。

イメージ図:

```text
┌─────────────────────────────────────────────────┐
│              レストランの厨房に例えると            │
└─────────────────────────────────────────────────┘

[deliver_now]  ホールスタッフが1人で全部やる
  注文取る → 料理する → 配膳する → 会計
  ↑ 料理してる間、他のお客さんを待たせる

[deliver_later]  役割分担する
  ホール: 注文取る → 配膳する → 会計（速い）
  キッチン (= worker): 料理に専念
  ↑ お客さんは注文後すぐ席に戻れる
```

## 2. このプロジェクトの構成

`docker-compose.yml` で4つのサービスが動いています:

| サービス | 役割 |
| --- | --- |
| db | MySQL データベース |
| backend | Rails API サーバー（リクエスト受付） |
| worker | Solid Queue ジョブワーカー（裏方） |
| frontend | Next.js 開発サーバー |

仕事の受け渡しの仕組み:

```text
[backend コンテナ]                [db コンテナ]               [worker コンテナ]
        │                              │                            │
        │ 「メール送って！」              │                            │
        │ をジョブとして登録             │                            │
        ├─────────────────────────────▶│                            │
        │                              │                            │
        │ レスポンス即返却              │ ジョブを定期的に            │
        │                              │ チェック ◀──────────────────┤
        │                              │                            │
        │                              │ 「メール送って」ジョブを渡す │
        │                              ├───────────────────────────▶│
        │                              │                            │
        │                              │                            │ メール送信
        │                              │                            │ 実行
        │                              │                            │
        │                              │ 完了したジョブを削除         │
        │                              │ ◀───────────────────────────┤
```

ポイント: Solid Queue は MySQL のテーブル をジョブの保管場所として使います。`solid_queue_jobs` などのテーブルがあり、そこにジョブの情報が書き込まれます。

## 3. deliver_now と deliver_later の違い

### deliver_now（同期）

```text
時間軸 →

ユーザー  ボタン押す ──┐
                       │
backend   DB更新 ──── メール送信中（3〜10秒） ──── レスポンス返す
                       │
                       └─ この間ユーザーは待たされる
```

### deliver_later（非同期）

```text
時間軸 →

ユーザー  ボタン押す ─┐
                      │
backend   DB更新 ── ジョブ登録 ── レスポンス返す（100ms以下）✨
                                       │
worker                                 └─ メール送信（3〜10秒）
                                          ↑ ユーザーには関係ない
```

## 4. 比較表

| 観点 | deliver_now | deliver_later |
| --- | --- | --- |
| レスポンス速度 | 遅い（3〜10秒） | 速い（100ms以下） |
| メールサーバー障害時 | リクエスト全体が失敗 | ジョブが自動リトライ |
| 大量送信 | サーバーが詰まる | キューで順番に処理 |
| インフラの複雑さ | シンプル（worker 不要） | worker が必要 |
| デバッグ | エラーがその場で見える | ジョブログを見る必要あり |
| 本番運用 | △ あまり推奨されない | ◎ 標準的な選択 |

## 5. なぜ deliver_later の方が良いのか（5つの理由）

### ① レスポンスが速い

メール送信は外部の SMTP サーバーと通信するので 3〜10秒かかることも普通。その間ユーザーを待たせるのは UX として最悪。

### ② 外部障害に強い

SendGrid や AWS SES などのメールサービスは たまに障害 が起きます。

- `deliver_now`: メール送信失敗 → API 全体が 500 エラー → ユーザーは「変更失敗した」と思う（実は DB は更新済み）
- `deliver_later`: メール送信失敗 → ジョブがリトライされるだけ → ユーザーには成功と表示

### ③ 自動リトライ

Solid Queue は失敗したジョブを 自動で再実行 してくれます。手動リトライを実装する必要がない。

### ④ サーバー資源の効率

- Rails のリクエスト処理枠（Puma の worker 数）は限られた資源。
- 1リクエストが10秒占有すると、その間他のユーザーをさばけません。
- worker に投げれば Rails 側はすぐ空く → より多くのリクエストをさばける。

### ⑤ タイムアウト事故を防げる

ロードバランサーや CDN には 30〜60秒のタイムアウト があります。SMTP が遅い日に運悪く引っかかると 504 エラーになることも。`deliver_later` なら絶対に引っかからない。

## 6. このプロジェクトの設計判断

CLAUDE.md より:

- ジョブキュー: Solid Queue（MySQL ベース）
- Redis 不使用 — ジョブキュー・セッション・キャッシュはすべて MySQL で管理。インフラをシンプルに保つ設計

一般的な構成との違い:

```text
[よくある構成]                  [このプロジェクト]
┌─────────┐                     ┌─────────┐
│  Rails  │                     │  Rails  │
└────┬────┘                     └────┬────┘
     │                               │
┌────▼────┐  ┌────────┐         ┌────▼────┐
│  MySQL  │  │ Redis  │         │  MySQL  │ ← 全部ここ
│  (DB)   │  │(キュー) │         │ DB+キュー│
└─────────┘  └────────┘         │+セッション│
                                └─────────┘
```

- メリット: Redis を立てなくて済む → インフラ構築・運用がシンプル
- トレードオフ: 高頻度ジョブ処理では Redis ベースより遅い（ただしメール送信用途なら十分）

## 7. いつ deliver_now を使うか

完全に禁止というわけではなく、こんな場面では `deliver_now` が使われます:

- RSpec などのテスト内 — その場で結果を確認したい
- rake タスクや管理スクリプト — そもそも worker が動いてないケース
- コンソール (`rails console`) での確認 — デバッグ目的

本番のリクエスト処理では基本 `deliver_later`、と覚えておけば OK。

## 8. まとめ

- ✅ メール送信のような時間のかかる処理は worker に任せる
- ✅ コードでは `.deliver_later` を呼ぶだけ
- ✅ ジョブの保管・実行は Solid Queue が MySQL でやってくれる
- ✅ ユーザーはレスポンスを待たされない
- ✅ 外部障害があっても自動リトライ

一言でいうと: 「メール送信は重い仕事だから、Rails 本体じゃなく worker に任せて、ユーザーには即レス返そう」 — これが `deliver_later` を使う理由です。
