## 概要

パスワード再設定 API は **「メアドが登録されているかどうか」を外部に漏らさない設計** にする必要がある。このプロジェクトでは **未登録メアドでも `success: true` を返す**ことで、レスポンスから登録の有無を判別できないようにしている。

## 脆弱性: ユーザー列挙攻撃（User Enumeration）

### 攻撃の流れ

```
[攻撃者]                                  [脆弱な API]
   │                                          │
   │ tanaka@example.com で再設定リクエスト    │
   ├─────────────────────────────────────────►│
   │  ◄── "登録されていません" ───────────────┤  ✗ スキップ
   │                                          │
   │ suzuki@example.com で再設定リクエスト    │
   ├─────────────────────────────────────────►│
   │  ◄── "メールを送信しました" ─────────────┤  ✓ 登録あり！
   │                                          │
   │ yamada@example.com で再設定リクエスト    │
   ├─────────────────────────────────────────►│
   │  ◄── "登録されていません" ───────────────┤  ✗ スキップ
   │                                          │
   ▼
[攻撃者が「サービス登録済みメアドリスト」を入手]
```

レスポンスに差異があると、攻撃者はメアドを順に試して **「登録されているメアド」だけを抽出できる**。

### 集められたメアドが招く二次攻撃

| 攻撃 | 内容 |
|---|---|
| **クレデンシャル・スタッフィング** | 他サービスから漏洩したパスワードリストを使って同じメアドでログイン試行 |
| **ブルートフォース** | 確定したメアドに対してパスワード総当たり |
| **フィッシング** | 「サービス利用者」と確定したターゲットに偽メール送付 |
| **個人情報の名寄せ** | 「この人はそのサービスを使っている」という事実自体が情報になる（主催者の身バレなど） |

## 対策: レスポンスを常に同一にする

### 危険な実装（やってはいけない）

```ruby
user = User.find_by(email: email)
return { success: false, errors: [{ message: "登録されていません" }] } if user.nil?

user.update!(reset_password_token: ..., reset_password_sent_at: ...)
PasswordResetMailer.password_reset_email(user).deliver_later
{ success: true, errors: [] }
```

- 登録済み → `success: true`
- 未登録 → `success: false`

→ **レスポンスから判別可能**

### 安全な実装（このプロジェクトの採用）

```ruby
def resolve(email:)
  errors = validate(email:)
  return { success: false, errors: errors } if errors.any?

  user = User.find_by(email: email)
  if user
    user.update!(
      reset_password_token: SecureRandom.urlsafe_base64(32),
      reset_password_sent_at: Time.current,
    )
    PasswordResetMailer.password_reset_email(user).deliver_later
  end

  { success: true, errors: [] }   # ← 常に success
end
```

| ケース | レスポンス | 実態 |
|---|---|---|
| 登録済みメアド | `success: true, errors: []` | メール送信する |
| 未登録メアド | `success: true, errors: []` | 何もしない |

**外から見ると同一**。攻撃者には区別できない。

## ユーザー体験への影響

| 観点 | 正直に返す設計 | 常に成功を返す設計（このプロジェクト） |
|---|---|---|
| 「未登録です」表示 | あり | なし |
| ユーザーが気づくきっかけ | レスポンスメッセージ | メールが届かない |
| セキュリティ | ❌ 登録メアドを列挙できる | ✅ 列挙不可 |
| 業界標準 | ❌ | ✅ |

- 正しいメアドなら → メールが届く → 再設定できる
- 間違ったメアドなら → メールが届かない → 「打ち間違えたかも」と気づく

**ユーザーには伝わるが、攻撃者には伝わらない** という構造。

## 他にも考慮すべき脆弱性

パスワード再設定フローは **アカウント乗っ取りの最大の攻撃経路** なので、列挙攻撃以外にも注意点がある。

| リスク | 対策 | このプロジェクトの現状 |
|---|---|---|
| メアド列挙攻撃 | 常に成功レスポンス | ✅ 対策済み |
| トークン推測 | 十分に長いランダム文字列 | ✅ `SecureRandom.urlsafe_base64(32)` |
| トークン再利用 | 成功時にトークンを nil クリア | ✅ 対策済み |
| トークン有効期限 | 短めに設定 | ✅ 2 時間 |
| メアドアカウント乗っ取り | 二段階認証など | ⚠️ メール認証の宿命 |
| 大量リクエスト（メール爆撃） | レート制限 | ❌ 未対策 |
| タイミング攻撃 | `User.find_by` の有無で応答時間が変わる可能性 | ⚠️ 未対策 |
| パスワード変更後のセッション無効化 | 他端末セッションを削除 | ❌ 未対策 |
| HTTPS の強制 | 本番でほぼ必須 | ⚠️ インフラ側で対応 |

## ポイント

- パスワード再設定 API のレスポンスは、登録済み / 未登録で**差分が出てはいけない**（列挙攻撃を防ぐため）。
- 安全な実装は「ユーザーが見つかればメール送信、見つからなければ何もしない」で、**どちらも `success: true` を返す**。
- 列挙攻撃で得られたメアドは、クレデンシャル・スタッフィング / フィッシング / 名寄せの起点になる。
- 正しいユーザーには「メールが届くかどうか」で結果が伝わるので、UX 上の損失は小さい。
- 列挙攻撃以外にも、トークン推測・再利用・有効期限・レート制限・タイミング攻撃・セッション無効化・HTTPS 強制など多面的な対策が必要。
